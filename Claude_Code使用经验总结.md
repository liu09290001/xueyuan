# Claude Code 使用经验总结

> 基于学员管理系统开发全过程的实践经验

---

## 一、需求理解

### 经验：PRD差距分析与需求拆解

**场景**：拿到PRD文档后，需要快速理解MVP需求与现有系统的差距。

**实践方法**：
1. 让Claude Code阅读PRD文档，自动提取功能点清单
2. 对比现有代码，生成差距分析报告（功能完成度、缺失项）
3. 按优先级（P0/P1/P2）分类待实现功能

**示例指令**：
```
请仔细阅读PRD文档，进行深度思考，完善PRD中要求的MVP和当前系统之间的差距，并重新review下代码
```

**收获**：
- 自动生成结构化的差距分析报告
- 功能完成度量化评估（如：功能65%，非功能40%）
- 明确P0优先级任务清单，避免遗漏关键需求

---

## 二、架构设计

### 经验：基于现有代码的增量设计

**场景**：在已有Flask项目基础上，需要新增考勤统计、权限控制等功能。

**实践方法**：
1. 让Claude Code先分析现有架构（models.py、routes.py、utils.py）
2. 基于现有模式设计新功能，保持代码风格一致
3. 遵循现有的API响应格式和错误码规范

**关键点**：
- 复用现有装饰器模式（如`@login_required`）扩展`@admin_required`
- 统计接口复用SQLAlchemy的聚合查询模式
- 前端图表复用Chart.js现有配置风格

**收获**：
- 新功能与现有代码无缝融合
- 避免引入新的技术栈增加复杂度
- 保持代码库的一致性和可维护性

---

## 三、编码实现

### 经验：分段编码与即时验证

**场景**：实现多个P0功能（考勤记录、状态切换、课程搜索等）。

**实践方法**：
1. 使用TodoWrite工具拆分任务，逐个实现
2. 每完成一个功能点立即标记完成
3. 修改代码时提供足够上下文，避免误改

**示例**：
```python
# 分段添加代码，而非一次性大量输出
# 先添加后端API
@api_bp.route('/students/<int:id>/status', methods=['PATCH'])
def update_student_status(id):
    ...

# 再添加前端交互
function changeStatus(id, status) {
    ...
}
```

**收获**：
- 小步快跑，降低出错风险
- 每个改动可追溯、可验证
- 避免大量代码一次性修改导致的冲突

## 四、Code Review

### 经验：自动化代码审查与问题定位

**场景**：Review现有代码，发现潜在问题和改进点。

**实践方法**：
1. 让Claude Code审查特定模块的代码质量
2. 识别安全隐患（SQL注入、XSS、权限漏洞）
3. 检查代码规范（PEP8、注释、错误处理）

**发现的问题示例**：
- 缴费金额未做类型转换，可能导致计算错误
- 删除操作缺少关联数据检查
- 部分接口缺少权限校验

**收获**：
- 快速定位代码问题
- 提供修复建议和代码示例
- 持续改进代码质量

## 五、测试设计与执行

### 经验：多维度测试用例自动生成

**场景**：需要设计功能测试、E2E测试、API测试、安全测试。

**实践方法**：
1. 根据PRD功能点自动生成测试用例
2. 分层设计：单元测试 → 接口测试 → E2E测试
3. 测试数据自动清理，避免脏数据污染

**测试类型覆盖**：
| 类型 | 用例数 | 覆盖范围 |
|------|--------|----------|
| 功能测试 | 30 | 5个模块核心功能 |
| E2E测试 | 15 | 完整业务流程 |
| API测试 | 60 | 全部接口 |
| 安全测试 | 10 | 认证/授权/输入验证 |

**关键代码**：
```python
# 测试后自动清理脏数据
def cleanup_test_data():
    test_students = Student.query.filter(
        Student.name.like('%测试%')
    ).all()
    for s in test_students:
        db.session.delete(s)
    db.session.commit()
```

**收获**：
- 测试覆盖率大幅提升
- 自动化执行，节省人工时间
- 测试报告自动生成（HTML/JSON）

## 六、安全测试

### 经验：安全测试用例设计与无害化验证

**场景**：验证系统的认证、授权、输入验证等安全机制。

**实践方法**：
1. 设计不产生脏数据的安全测试用例
2. 使用查询操作验证SQL注入防护
3. 使用拒绝场景验证输入校验

**安全测试要点**：
```python
# 认证测试 - 不产生脏数据
t.test("认证", "未登录拒绝访问", lambda: session.get(
    f"{BASE_URL}/api/students").status_code == 401)

# 输入验证 - 使用查询而非写入
t.test("输入验证", "SQL注入防护", lambda: session.get(
    f"{BASE_URL}/api/students",
    params={"keyword": "'; DROP TABLE students;--"}).status_code == 200)

# 授权测试 - 验证拒绝场景
t.test("授权", "未登录访问统计拒绝", lambda: new_session.get(
    f"{BASE_URL}/api/statistics/dashboard").status_code == 401)
```

**收获**：
- 安全测试不污染生产数据
- 覆盖OWASP常见安全风险
- 验证系统安全机制有效性

## 七、版本管理与数据清理

### 经验：测试数据生命周期管理

**场景**：测试过程中产生大量脏数据，需要及时清理。

**实践方法**：
1. 测试用例设计时考虑数据清理
2. E2E测试结束时自动清理测试数据
3. 定期检查并清理遗留脏数据

**数据清理策略**：
```python
# E2E测试自带清理步骤
def scenario_student_lifecycle(tester):
    # ... 测试步骤 ...
    # 最后清理
    tester.session.delete(f"{BASE_URL}/api/students/{student_id}")
    steps.append("5. 删除学员")

# 批量清理脏数据
def cleanup_all_test_data():
    # 按依赖顺序删除：考勤 → 缴费 → 报名 → 学员/课程
    for sid in test_student_ids:
        Attendance.query.filter_by(student_id=sid).delete()
        Enrollment.query.filter_by(student_id=sid).delete()
        Student.query.filter_by(id=sid).delete()
    db.session.commit()
```

**收获**：
- 测试环境保持干净
- 避免脏数据影响后续测试
- 数据库状态可控可追溯

## 九、踩坑与解决方案

### 1. 服务器未启动导致测试失败

**问题**：执行测试时报 `ConnectionRefusedError`，测试全部失败。

**原因**：Flask服务器未启动，测试脚本无法连接到API。

**解决**：
- 测试前确认服务器已启动（`python run.py`）
- 可考虑在测试脚本中添加连接检测和友好提示

---

### 2. API限流导致测试失败

**问题**：安全测试中认证相关用例大量失败，返回429状态码。

**原因**：登录接口有 `@rate_limit(limit=10, window=60)` 限流装饰器，短时间内多次登录触发限流。

**解决**：
```python
# 测试用例间添加延时
time.sleep(1)  # 单个测试间隔

# 测试套件间添加较长等待
print(">>> 等待60秒避免限流...")
time.sleep(60)
```

---

### 3. Edit工具多匹配报错

**问题**：使用Edit工具时报错 `Found 3 matches of the string to replace`。

**原因**：要替换的字符串在文件中出现多次，工具无法确定替换哪一个。

**解决**：
- 提供更多上下文，使匹配字符串唯一
- 或使用 `replace_all=true` 替换所有匹配项
- 复杂情况下使用Write工具重写整个文件

---

### 4. 测试产生脏数据污染数据库

**问题**：安全测试中XSS和超长字符串测试产生了脏数据（如 `<script>alert(1)</script>`）。

**原因**：测试用例设计时未考虑数据清理。

**解决**：
- 安全测试改用查询操作验证，不产生写入
- E2E测试每个场景结束时清理测试数据
- 测试完成后统一执行清理脚本

---

### 5. Windows路径问题

**问题**：Bash命令中使用 `D:\path` 格式路径报错。

**原因**：Git Bash环境下需要使用Unix风格路径。

**解决**：
```bash
# 错误
cd D:\Claude-test\xueyuan-test

# 正确
cd /d/Claude-test/xueyuan-test
```

---

### 6. 中文乱码显示

**问题**：测试输出中中文显示为乱码（如 `ѧԱ����`）。

**原因**：Windows控制台编码与Python输出编码不一致。

**影响**：不影响测试结果，仅影响显示。

**建议**：
- 测试报告使用HTML/JSON格式，避免控制台乱码问题
- 或设置控制台编码为UTF-8

---

### 7. 重复询问用户确认

**问题**：每次清理脏数据都询问用户确认，效率低下。

**解决**：
- 明确约定后，测试产生的脏数据自动清理
- 减少不必要的交互，提升效率
